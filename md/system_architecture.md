# Chameleon 시스템 아키텍처 전체 도식

> 이 문서는 Chameleon 제품의 전체 시스템을 한 눈에 파악하기 위한 아키텍처 명세입니다.
> 데이터의 흐름 — 수집 → 해석 → 가공 → 응답 — 을 레이어별로 구조화합니다.


파일 구성 요약

  총 11개 섹션으로 전체 시스템을 커버합니다:

**0. 전체 시스템 조감도**: 외부 세계와 쇼핑몰, Chameleon 서버 및 DB 간의 유기적 구조도
**1. 데이터 수집 파이프라인**: 카페24 API를 통한 원본 데이터 수집부터 LLM 정규화, 임베딩, Supabase 저장까지의 흐름
**2. 유저 진입 & 신호 수집**: 유저의 유입 경로 및 행동 패턴을 Level 1~4 계층으로 분류하여 처리하는 전략
**3. 프론트엔드 UI**: 사이드바 iframe 구조, 채널톡과의 UI 공존 및 모바일 대응 전략
**4. 핵심 처리 엔진**: 이해 Agent와 검색 Agent를 분리하여 유저 의도를 분석하고 최적의 LLM 응답을 생성하는 핵심 레이어
**5. 유저 프로파일 & 학습 루프**: 대화 턴마다 프로파일을 실시간 업데이트하고 구매 전환 피드백을 통해 추천 품질을 높이는 알고리즘
**6. PDP 개인화**: 유저 세그먼트 및 의도에 맞춰 상품 상세 페이지의 DOM을 동적으로 조작하는 방식
**7. End-to-End 데이터 플로우**: 최초 유입부터 응답까지의 전체 타임라인과 응답 속도 최적화 목표
**8. 기술 스택**: 사용된 언어, 프레임워크, 인프라 및 각 기술을 선택한 전략적 이유
**9. 한국 시장 특화**: 네이버/카카오 등 로컬 플랫폼 생태계 연동 및 특화 기능
**10. 우선순위 로드맵**: 핵심 기능(P0)부터 전략적 차별화 포인트(P3)까지의 단계적 개발 계획
**11. 시스템 의존성 지도**: 모듈 간의 선후 관계와 병목 지점 파악을 위한 종속성 구조도

## 2. 유저 진입 & 신호 수집 (Signal Collection)

> "유저가 들어오는 순간 무슨 일이 일어나는가"

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      USER ENTRY & SIGNAL DETECTION                      │
│                         (< 수십 밀리초 내 완료)                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  유저가 브랜드 쇼핑몰 URL 접속                                          │
│         │                                                               │
│         ▼                                                               │
│  widget.js (Scripttag) 실행                                             │
│         │                                                               │
│         ├─── LEVEL 1: 유입 컨텍스트 신호 (즉각, 가장 강력)              │
│         │    ├─ URL 파싱: utm_campaign, utm_content, utm_source         │
│         │    ├─ document.referrer 분석                                  │
│         │    │   ├─ chatgpt.com → 🔴 AI 트래픽 (고관여, 구매 의도 강함) │
│         │    │   ├─ google.com  → 일반 검색 트래픽                      │
│         │    │   ├─ instagram.com → 광고 유입 트래픽                    │
│         │    │   └─ (없음) → 직접 방문                                  │
│         │    └─ 커스텀 파라미터: k_seg=runner, k_seg=gift_buyer 등      │
│         │                                                               │
│         ├─── LEVEL 2: 온사이트 행동 신호 (수동, 실시간 관찰)            │
│         │    ├─ 스크롤 깊이: 어느 섹션(소재/사이즈/배송)에서 멈추는가    │
│         │    ├─ 섹션 체류 시간: 가격 vs 소재 설명 중 어디를 오래 보는가  │
│         │    └─ 이미지 클릭 패턴: 라이프스타일 컷 vs 디테일 컷 선호       │
│         │                                                               │
│         ├─── LEVEL 3: 능동적 대화 신호 (가장 정확, 유저가 직접 말함)    │
│         │    └─ AI 채팅에서 유저가 입력하는 자연어 (메인 인텐트 소스)     │
│         │                                                               │
│         └─── LEVEL 4: 컨텍스트 & 이력 신호                             │
│              ├─ localStorage: 이전 방문 상품 카테고리, 대화 기록          │
│              ├─ 시간/요일: 야간(충동구매?), 주말(선물?)                  │
│              └─ 디바이스: 모바일(빠른 결정) vs 데스크탑(비교 쇼핑)        │
│                                                                         │
│         ▼                                                               │
│  ┌────────────────────────────────────────────────────────────────┐     │
│  │  Signal → Chameleon API 전송                                    │     │
│  │  POST /api/classify                                             │     │
│  │  { referrer, utm, device, localStorage, currentPage }          │     │
│  └────────────────────────────┬───────────────────────────────────┘     │
│                               │                                         │
│                               ▼                                         │
│  초기 세그먼트 판정 (LLM/ML)                                                │
│  → "Gift Buyer", "Spec-focused Runner", "Price Sensitive" 등            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 4. 핵심 처리 엔진: Intent 분석 → 검색 → 응답 생성

> "백엔드에서 무슨 일이 일어나는가" (가장 중요한 레이어)

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         CORE PROCESSING ENGINE                          │
│                                                                         │
│  유저 메시지 수신: "남자친구 생일 선물인데 오버사이즈 좋아해요"         │
│                                                                         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  AGENT 1: 이해 에이전트 (Understanding Agent)  ← 검색은 하지 않음      │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│                                                                         │
│  INPUT:  유저 메시지 + 대화 히스토리(최근 4턴) + 유저 프로파일          │
│                                                                         │
│  PROCESS: LLM 구조화 출력                                               │
│  {                                                                      │
│    situation:   "선물",                          ← 자기구매가 아님      │
│    recipient:   "남자친구",                       ← 남성용 상품 필터     │
│    occasion:    "생일",                           ← 특별함 필요         │
│    style:       ["오버사이즈", "캐주얼"],                                │
│    priceRange:  { min: null, max: null },                               │
│    constraints: [],                                                     │
│    budgetType:  "flexible",                      ← "가격 언급 없음"      │
│    searchQuery: "남성 오버사이즈 캐주얼 셔츠 선물용"                     │
│    clarifyNeeded: false                          ← 질문 필요 판단       │
│  }                                                                      │
│                                                                         │
│  ※ clarifyNeeded: true이면 → 질문 1개만 던지고 대기                   │
│     "남자친구 분 평소 사이즈가 어떻게 되세요?"                          │
│                                                                         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  AGENT 2: 검색 에이전트 (Retrieval Agent)  ← 이해는 하지 않음          │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│                                                                         │
│  STEP A: Semantic Search (벡터 검색)                                    │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  searchQuery → Gemini embedding → 쿼리 벡터                     │  │
│  │  pgvector cosine similarity 계산                                 │  │
│  │                                                                 │  │
│  │  "오버사이즈 캐주얼" ────────▶ [0.23, -0.11, ...]               │  │
│  │                                        ↕ 유사도 계산            │  │
│  │  상품 5534 임베딩  [0.12, -0.34, ...] → similarity: 0.61       │  │
│  │  상품 5533 임베딩  [0.25, -0.09, ...] → similarity: 0.95 ✓     │  │
│  │                                                                 │  │
│  │  → 상위 20개 후보 추출                                           │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  STEP B: Keyword Filter (SQL 필터)                                      │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  WHERE tags @> ARRAY['남성']                                     │  │
│  │    AND (p_max_price IS NULL OR price <= p_max_price)            │  │
│  │    AND NOT (tags @> ARRAY['여성전용'])                           │  │
│  │    AND NOT (normalized_data->>'NOT' @> '["오버사이즈"]')         │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  STEP C: Re-ranking  (이진법 배제 아닌 점수화)                          │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  후보 상품 5개 → LLM에게:                                       │  │
│  │  "이 상황(선물, 오버사이즈)에 가장 맞는 순서로 재정렬하고        │  │
│  │   왜 이 제품이 적합한지 이유를 생성해줘"                         │  │
│  │                                                                 │  │
│  │  ※ 예산 초과여도 "최적 조건"이라면 순위에 포함                  │  │
│  │    → 이유 설명: "예산보다 조금 높지만 핏이 완벽해요"             │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  LLM 응답 생성 (Response Generator)                                     │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│                                                                         │
│  PROMPT:                                                                │
│  "당신은 [브랜드명] 전문 스타일리스트입니다.                            │
│   절대로 관련 없는 상품을 추천하지 마세요.                              │
│   이 상황에 맞는 이유를 구체적 수치로 설명하세요."                      │
│                                                                         │
│  OUTPUT (JSON):                                                         │
│  {                                                                      │
│    recommendations: [                                                   │
│      {                                                                  │
│        product_no: 5533,                                                │
│        reason: "가슴 여유 39cm라 확실한 오버사이즈예요",                  │
│        tradeoff: null                                                   │
│      }                                                                  │
│    ],                                                                   │
│    answer: "베이지 체크 패턴에 가슴 여유가 39cm라 ...",                  │
│    followUp: "남자친구 분 평소 사이즈가 어떻게 되세요?"                   │
│  }                                                                      │
│                                                                         │

## 7. 전체 데이터 플로우 (End-to-End)

> "처음부터 끝까지 데이터가 어떻게 흐르는가"

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

사전 준비 (오프라인)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

카페24 API                LLM 정규화              Supabase
(상품 원본 데이터)    →    (HTML 파싱 + 수치 도출)   →  (products + embeddings)
       매일 새벽 자동 실행 (증분 업데이트)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

실시간 (유저가 사이트에 있는 동안)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

① 유저 진입
   URL + referrer + UTM
         ↓
② Signal 수집 & 초기 세그먼트 판정 (<100ms)
   "Gift Buyer" / "Price Sensitive" / "AI Traffic" 등
         ↓
③ PDP 즉시 개인화 (DOM 조작)
   세그먼트에 맞는 배지 + FAQ 칩 주입
         ↓
④ 사이드바 AI 채팅 활성화
   유저가 자연어로 질문 입력
         ↓
⑤ AGENT 1: Intent 분석 (~200ms)
   JSON 구조화 출력 (situation, style, constraints, searchQuery...)
         ↓
⑥ AGENT 2: Hybrid Search (~300ms)
   벡터 검색(20개 후보) → SQL 필터 → Re-ranking(최종 3~5개)
         ↓
⑦ LLM 응답 생성 (~500ms)
   추천 이유 + Trade-off + Follow-up 질문 포함
         ↓
⑧ 프론트엔드 렌더링
   상품 카드 3개 + 대화 메시지 표시
         ↓
⑨ 유저 행동 추적
   클릭/구매 여부 → User Profile 업데이트 → 학습 루프

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

총 응답 시간 목표: ~1초 이내
(Signal: <100ms | Intent: ~200ms | Search: ~300ms | LLM: ~500ms, 병렬 가능)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> 지금 당장 짜야 할 것부터, 나중에 붙일 것까지

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔴 P0: 핵심 작동 (없으면 제품이 존재하지 않음)  / Week 1~5
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ① 상품 수집 파이프라인 (Phase 1)

  ② Intent 분석 엔진 (Phase 2)

  ③ 벡터 검색 + Hybrid Re-ranking (Phase 3)
     searchQuery → 상위 3~5개 상품 추출
     WHY: 이게 없으면 추천 결과가 나오지 않음

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🟠 P1: 차별화 가치 (있으면 월등히 나아짐)  / Week 6~8

     "예산 5천원 초과지만 오버사이즈 여유가 10cm 더 큼"
     WHY: 이게 있어야 "나를 이해해줬다"는 경험이 발생

  ⑤ 사이트 전역 어시스턴트 UI (Phase 5)
     iframe 사이드바 + 페이지 이동 간 컨텍스트 유지
     WHY: 유저가 실제로 쓰는 인터페이스

  ⑥ PDP 세그먼트별 개인화
     DOM 조작으로 배지 + FAQ 칩 + 설명 텍스트 교체
     WHY: 채팅 안 쓰는 유저에게도 개인화 가치 전달

🟡 P2: 깊이와 정확도 (정밀도를 높임)  / Week 9~12

  ⑦ 유저 프로파일 축적 (Phase 4)
     대화 히스토리 → 프로파일 점진적 업데이트
     WHY: 재방문 시 처음부터 묻지 않아도 됨

  ⑧ Silent Abandonment 측정 대시보드
     이탈 이벤트 → 의도 불통 감지 → 브랜드 리포팅
     WHY: Chameleon의 ROI를 브랜드에게 증명하는 핵심 지표

  ⑨ 브랜드별 AI 페르소나 커스터마이징
     브랜드 가이드라인 학습 → 톤앤매너 + 추천 기준 차별화
     WHY: "파타고니아 AI vs 럭셔리 브랜드 AI"를 다르게 만드는 것

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ⑩ AI 트래픽 (GEO) 특수 처리
     chatgpt.com referrer 감지 → "고관여 진입" 모드
     WHY: AI에서 넘어온 유저는 이미 구매 의도가 강함
     (이 레이어가 있어야 GEO ↔ On-site 이중 전략 완성)

  ⑪ 브랜드 CRM 데이터 환류
     Chameleon 수집 인텐트 데이터 → 브랜드 CRM 연동
     WHY: 브랜드가 고객 관계의 소유권을 유지하게 됨

  ⑫ 이미지 → Vision 모델 정규화
     description이 없거나 부실한 상품을 이미지로 분석
     WHY: 상품 데이터 품질이 낮은 브랜드도 커버 가능
